# Self VM Development Skill

You are working on the Self language VM â€” a research virtual machine originally from Sun Labs, now maintained as an open-source project. The active target is **vm64** (64-bit, interpreter-only on aarch64).

## Project Layout

```
vm64/                    # 64-bit VM (active target)
  src/
    any/                 # Platform-independent code
      asm/               # Assembler/code generation
      fast_compiler/     # NIC (non-inlining compiler) â€” NOT active on aarch64
      interpreter/       # Bytecode interpreter (the active execution engine)
      lookup/            # Method lookup, inline caches, send descriptors
      memory/            # Object memory, GC, generations, spaces
      objects/           # oop/map hierarchy (smiOop, memOop, slotsMap, etc.)
      os/                # Platform windows (quartzWindow, xlibWindow)
      parser/            # Scanner, parser, bytecodes
      prims/             # Primitives and glue
      runtime/           # Process switching, frames, stack, debug, NLR, shell
      sic/               # SIC (simple inlining compiler) â€” NOT active on aarch64
      tests/             # VM unit tests
      zone/              # Code zone, nmethods, stubs
    amd64/               # x86_64-specific code (headers + some .cpp)
    aarch64/             # ARM64-specific code (mostly symlinks to amd64/)
    unix/                # Unix-specific OS layer
    cocoa/               # Experimental Cocoa console (not used in standard builds)
  cmake/                 # Build configuration (some symlinked from vm/cmake/)
  build_support/         # includeDB, vmDate template, icons, plists
vm/                      # 32-bit VM (legacy, shares cmake modules)
objects/glue/            # Primitive glue code generated by primitiveMaker
```

## Key Architecture Facts

- **No JIT on aarch64/x86_64**: `FAST_COMPILER` and `SIC_COMPILER` are NOT defined. All execution goes through the interpreter. Code guarded by `#if defined(FAST_COMPILER) || defined(SIC_COMPILER)` is dead.
- **aarch64 files are mostly symlinks** to amd64 counterparts. Edit the amd64 original.
- **Process switching** uses cooperative green threads. `SetSPAndCall` in `runtime_stubs_aarch64.cpp` is the assembly heart â€” it saves/restores callee-saved registers and switches stacks. `processSemaphore` guards critical sections across transfers.
- **NLR (non-local return)** on 64-bit interpreter uses `longjmp` (`interpreter_longjmp_for_NLR`) since `ContinueNLRFromC` doesn't work without JIT.
- **Headers lack include guards**. The precompiled header (`_precompiled.hh`) is generated by `makeDeps` from `includeDB`. Double-inclusion causes redefinition errors.
- **`config.hh`** defines all architecture/OS/compiler constants. On macOS it `#undef`s and redefines `TARGET_ARCH` from compiler built-ins.
- **`types.hh`** defines the integer types (`fint`, `smi`, `int32`, etc.) and the oop/map class hierarchy.
- **Quartz windows** on aarch64 use `quartzWindow.mm` (Objective-C++/Cocoa/AppKit). It must import system frameworks BEFORE Self headers to avoid name collisions (e.g., `const int32 K` vs `NSDictionary<K,V>`).

## Build System

The build uses cmake. Key options for vm64 on macOS:

```
cmake -G Xcode -S . -B cmake-build-xcode-vm64 \
  -DBUILD_64BIT=ON -DSELF_QUARTZ=ON -DSELF_X11=ON -DSELF_XFT=ON
```

After generating or regenerating a build directory, always run:
```
cmake -P <build-dir>/vm64/configureVmDate.cmake
```

### Xcode-specific gotchas
- The Xcode generator doesn't use `GCC_PREFIX_HEADER` (blanket PCH) because it would inject the PCH into `.mm` files, breaking include ordering. Instead, per-file `-include` flags are applied via `NEED_PCH_EXT` filtering in `mac_osx.cmake`.
- `SELF_OSX_COCOA=ON` is incompatible with Quartz windows â€” it runs the VM on a background thread, but macOS requires UI on the main thread. Don't enable it with `SELF_QUARTZ`.
- Xft requires the freetype2 include directory and explicit linking of `libXft` and `libfreetype` (cmake's `FindX11` doesn't add them automatically).

## Debug vs Release

- **Debug**: `GENERATE_DEBUGGING_AIDS=1`, `GENERATE_ASSERTIONS` defined, no optimization. Assertions are active â€” many latent bugs surface only here.
- **Release**: `GENERATE_DEBUGGING_AIDS=0`, assertions compiled out, optimized. Use for running snapshots (opening desktop, etc.).
- Use `RelWithDebInfo` for optimized builds with debug symbols and assertions.

## Common Assertion Patterns

When an assertion fires in Debug that doesn't fire in Release:
1. Check whether it's a genuine bug or a stale assertion from I386/SPARC days
2. Many assertions reference `SaveOutgoingArgumentsOfPatchedFrames` which is `false` on x86_64/aarch64 but assertions assume it's `true` (I386 relic)
3. Guard architecture-specific assertions with `#if TARGET_ARCH == I386_ARCH`
4. The `processSemaphore` assertion in `Process::transfer()` can fire when `longjmp`-based NLR skips cleanup â€” check `nlrSupport.cpp`

## Debugging Tips

- Set watchpoints on `processSemaphore` to find who left it set
- The `printP()` macro traces process transfers when `traceP` is true
- `traceV` enables verbose VM tracing
- `WizardMode = true` enables detailed crash diagnostics
- When tracing optimized code, tag every trace line with method identity â€” multiple OptimizedActivations run interleaved
- `~/.lldbinit` must pass Self's signals through: SIGALRM, SIGVTALRM, SIGUSR1, SIGUSR2, SIGIO, SIGWINCH (all `--stop false --notify false`)
- `quartzWindow.mm` sets an empty main menu before `[NSApp finishLaunching]` to prevent AppKit from lazy-loading Writing Tools and other frameworks â€” without this, `dyld`'s `lldb_image_notifier` causes multi-second stalls under the debugger
